    Phase 2 Rule Abstraction

    Going through the initial rule cleanup for phase 2 went pretty smoothly this time around. I give the credit there to my categorized abstraction catalog and the fact I named them. I did run into a problem where I hadn't updated the names of some abstractions that I changed partway through my work on phase 1 (or wrote out the name differently for the same symbols), but that was relatively minor.

    I did make the choice to not make multiple sets of rules for abstractions with multiple variables (i.e. \(\{x, y, z\}\)). Instead, I left it with just substituting one variable for Chi, \(X\). For of the abstractions where I had created multiple rules, I only had one variable that actually changed. For example, \(\abstraction{A_\oplus\cap B_\oplus}\) had five rules created for it, but they all only varied by the value of \(y\) - \(x\) was always \(0\). As such, I created one set of rules substituting \(X\) for \(y\), and left it at that. As I encounter situations requiring different values of the remaining variables, I'll decide whether to account for those by substituting the other variable or by just making making another set with the another static value.

    One struggle I did encounter, probably from the naming issue, was that I missed some rules along the way. As such, I'm not exactly sure what number they originally were leading to several rule references that are like \(\rule{??}\) or \(\rule{51?)\). I need to perform a test run of the walkthrough and double check the references and update those uncertainties.

    Freedom!!!

    I finally completed the phase 1 section last night. I so utterly despise that part of the game because it's just repeating the same abstractions over and over again. At least with even one variable, I can cut down the number of individual rules I have to create.

    Final Counts

    I ended up with seventy-nine rules for thirty-seven abstractions. That feels way too high. Granted, I did try to split up some four-region abstractions into smaller ones, but whole "hybrid" abstractions section feels... cludgey?

    Organization

    One positive from completing this phase is that I was able to subdivide the abstractions page some more. I added sections for "reductions" (turning complex regions into less complex ones) and "intersections" (identifying a clue for the intersection of regions). This led me to deciding to use how I'm naming abstractions to help me categorize them, as all the rules in a given section should have similar symbolic names.

    That led to the "substraction" category. At first, I questioned if I should separate the proper subset rules (e.g. \(A_{(A\subsetneq B)_\bot}\)) from complement rules (e.g. \(A_{(A_\setminus B)_\top}\)). I ended up deciding not to go that around, since they are both mirroring the same idea, subtracting one region from another; it just happens that in come cases, the region we're removing is a proper subset.

    On the other hand, there were several abstractions that still felt tacked on. Taking this symbolic categorization approach showed me why - they're hybrids! All of them deal with some complement where one side is some product of two other regions, typically intersections (e.g \(A_{((A_1\cap{B_{\leq1}})\setminus C_\geq)_\bot}\)). Those are now in their own category.

    Descent into Madness...

    Can I just start off by saying... I hate phase 1 of this game.

    Having to solve everything by static rules is so mindnumbingly tedious. Whereas phase 0 can be completed with ten rules, I'm now up to thirty-eight, and I'm only two-thirds of the way to the level goal! What's more, I wouldn't be surprised if I must double that number for the next level set.

    Rule Convolution

    The second column of level sets can lead to some rule creep simply because you now have to account for abstractions that work the same way for both constant values and their minimum/maximum counterparts. The third column, however, is that times ten -- well, times five in one sense and potentially times 120 depending on how many variations of "or" rules are required.

    Special-Case Rules

    Still, I am mostly happy with this abstraction-library approach and the specific abstractions I've defined so far. There's only one that seems too specific to me, \(A_{A\setminus B_2}\). I've played around with it, and I can either parameterize one region or the other, but not both. There are a few abstractions that specifically require single-bomb regions, but I feel those are the exception to my concerns.

    A solution may come to me as some point in the future, but at least it's only one special case so far.

    Abstraction Categorization

    One other things I'm concerned about is how I categorize my abstractions. Currently, I have three categories: basic, wires, and subtraction. The problem is, the substraction category is getting quite large.

    Most of them fit a pretty clear pattern - various variation of subtracting one region from another. I can model them in a pretty clear table. Unfortunately, that leaves the various three-region abstractions. I feel like there could be a pattern similar to the two-region abstractions, but the ones I've made so far don't seem to be following it.

    I guess I'll wait and see, but I do feel I need split that category before I move beyond phase 1.

    Doing a Lot with a Few

    I have started on the walkthrough and completed "phase 0" tonight. Going through the process again (and for maybe the fourth time overall), I'm really happy with the approach I'm taking in describing rules to create. I'm taking an approach of observe-abstract-solidify:

    1. Observe. Being a walkthough, the emphasis is on solving a particular level. As such, I have a rule that I'm considering for a very specific case.
    2. Abstract. Once I have a working "proto-rule", I look at how I can abstract away the static numeric values. This creates a function that I can then plug values into to create a rule.
    3. Solidify. With those to pieces, I can specify which values would go into the abstraction variables to produce a rule effectively matching the observation case(s).

    It is those "solidified" rules that I actually create in the game. I show all three parts in the walkthrough so the reader can hopefully connect what I'm seeing in the level into the rule I'm finally suggesting.

    I was taking an approach like this in my last attempt at a walkthrough, but a oddly large change I made with this version is keeping a separate page for all the abstractions and naming each abstraction. I'm trying to make the names a descriptively-dense symbolic summary of the rule. For example, the abstration effectively defining an empty region is \(A_\bot\) -- the abstraction of "false" -- and the one defining a full set is \(A_\top\) - the abstraction of "true". The abstraction of a rule for two regions where the subtraction of a region \(B\) from region \(A\) that's full of bombs is \(A_{{A/B}_\top}\). This both gives me something to refer to for an abstration rather than just the function itself and makes me consider precisely what it's abstrating.

    Having this library of abstrations also helps me review what abstractions I've already made and reuse them rather than looking at what rule I could make, make an abstraction, and compare it to the previous rules in the walkthrough to see if I've already used that one. It really felt like abstractions were an afterthought; whereas now, they're a starting point.

    Fun facts, there are only four abstractions needed to complete the tutorial levels -- our two "definition" abstractions and their respective mappings to subtracting two regions.

    \[ \begin{array}{c|cc} & \text{Full }(\top) & \text{Empty }(\bot) \\ \hline \text{Basic} & A_\top & A_\bot \\ \text{Subtraction }(A/B) & A_{{A/B}_\top} & A_{{A/B}_\bot} \end{array} \]
